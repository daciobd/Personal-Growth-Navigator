# Auditoria Jornada-EU
**Objetivo:** confirmar que o que o app mostra, o que o banco grava e o que o admin lê são a mesma história.

---

## Como usar este roteiro

Para cada cenário: execute os passos, registre o resultado (✅ OK / ❌ Falhou / ⚠️ Parcial) e anote o que viu.

Você vai precisar de:
- Acesso ao app (`app.jornada-eu.com.br`)
- Acesso ao LONGEVI/admin
- Acesso ao banco (Replit ou psql direto)

---

## Cenário 1 — Usuário novo sem histórico

**Perfil:** primeira vez no app, sem nenhuma interação anterior.

| # | Passo | O que verificar | Resultado |
|---|-------|-----------------|-----------|
| 1 | Criar conta nova | Onboarding inicia sem erro | |
| 2 | Completar o fluxo de entrada | `segmentSource = none` no banco | |
| 3 | Carregar o plano | Evento `plan_loaded` aparece **uma única vez** no admin | |
| 4 | Verificar badge | Nenhum badge exibido | |
| 5 | Verificar no admin | Tab "App" mostra o usuário; `done = 0` | |

**Critério de aprovação:** 1 evento `plan_loaded`, `segmentSource = none`, sem badge, sem duplicidade.

## Cenário 2 — Usuário recorrente com histórico pessoal

**Perfil:** usuário que já fez check-ins anteriores, tem dados de Big Five estimados.

| # | Passo | O que verificar | Resultado |
|---|-------|-----------------|-----------|
| 1 | Entrar com conta existente | App carrega estado anterior corretamente | |
| 2 | Abrir o plano | `segmentSource = personal` no banco | |
| 3 | Verificar metadata do evento | Valor de `segmentSource` aparece em metadata no admin | |
| 4 | Checar conversão nos tabs | Conversão aparece corretamente nos tabs novos do admin | |

**Critério de aprovação:** `segmentSource = personal` gravado e visível no admin com metadata correto.

## Cenário 3 — Usuário que entra e sai várias vezes no mesmo dia

**Perfil:** abre o app, fecha, abre de novo — múltiplas vezes no mesmo dia.

| # | Passo | O que verificar | Resultado |
|---|-------|-----------------|-----------|
| 1 | Abrir app → fechar → abrir 3x | Apenas **1** evento `plan_loaded` no dia | |
| 2 | Verificar banco | Sem linhas duplicadas para o mesmo `user_id` + `date` | |

**Critério de aprovação:** idempotência — múltiplas entradas não geram duplicidade de eventos.

---

## Cenário 4 — Usuário marca done e tenta mudar

**Perfil:** completa uma prática, depois tenta marcar outra resposta.

| # | Passo | O que verificar | Resultado |
|---|-------|-----------------|-----------|
| 1 | Completar uma prática | `done = true` gravado no banco | |
| 2 | Tentar tocar em outra opção | UI bloqueia sem travar | |
| 3 | Usar "Corrigir resposta" | Reseta o estado corretamente | |
| 4 | Confirmar nova resposta | Banco atualiza; não duplica o evento | |

**Critério de aprovação:** bloqueio funciona, "Corrigir resposta" é o único caminho para resetar.

---

## Cenário 5 — Usuário offline com sync posterior

**Perfil:** perde conexão durante uso, reconecta depois.

| # | Passo | O que verificar | Resultado |
|---|-------|-----------------|-----------|
| 1 | Entrar no app com conexão | Estado carregado normalmente | |
| 2 | Desligar Wi-Fi/dados | App não crasha | |
| 3 | Fazer uma interação offline | UI responde (mesmo que com estado local) | |
| 4 | Reconectar | Sync ocorre; `source` do evento é preservado | |
| 5 | Verificar banco após sync | Sem duplicidade; dados corretos | |

**Critério de aprovação:** `source` preservado, sem duplicidade após reconexão.

## Verificações cruzadas no admin (LONGEVI)

Após rodar os 5 cenários, verificar estas combinações no admin:

| Combinação | O que checar |
|------------|--------------|
| `segmentSource × conversion` | Usuários `personal` convertem mais que `none`? |
| `source × done rate` | Taxa de conclusão varia por source? |
| `adaptationLevel × segment` | Níveis de adaptação distribuídos por segmento fazem sentido? |
| `markerKey × adaptationLevel × 1+/3+/7+` | Retenção por marcador e nível está coerente? |
